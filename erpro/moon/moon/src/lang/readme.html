关于功能屏蔽开放模式

台湾不采用该模式

针对海外（韩国，越南，泰国）平台采用功能屏蔽开放模式，针对某个版本，运营的负责人与策划方进行商讨，合理划分功能块，

进行分“期”处理。此后根据平台需求逐步更新（开放）对应的“期"的功能块。为了有效的进行屏蔽开放工作，应根据分“期”情况，

从最后一期开始逐期进行屏蔽，在版本库中，每期屏蔽完成应进行一次commit，并做好详细的注释。后续功能开放时，反向操作即可。

关于服务端版本库维护

  • 台湾
  • 韩国/越南/泰国

版本维护 —— 台湾

    台湾采用了4个分支，分别是, master_track_tw, tw_next_develop, tw_master, tw_develop

  • master_track_tw
    • 该分支提供三个功能，第一：从国服 master 上去版本, merge 相应的tag；第二：针对台湾版本做一些持久化修改，比如去掉敏感词库, 开启跨服聊天, 屏蔽强化祝福等；第三：作为 tw_next_develop 分支的来源
  • tw_next_develop
    • 该分支立足于：预备当前规划的下一次的版本更新， 该分支需要从 master_track_tw 上取下一次需要更新的版本。且应该在该分支上做只针对当前版本需要做的特殊化处理，所以建议每次版本更新采用覆盖的方式, 即：每次版本更新先清空该分支下文件，再拷贝 master_track_tw 上的文件到该分支下。该分支对应测试服的NEXT服。
  • tw_develop
    • 该分支相当于国服的 develop，作为针对线上服的测试服。每次版本更新应该 merge 分支 tw_next_develop。该分支对应测试服的 0 服
  • tw_master
    • 该分支相当于国服的 master，直供台湾线上服，新开服。每次版本更新应该 merge 分支 tw_develop。

版本维护 —— 韩国/越南/泰国

    这些国家采用了5个分支，分别是, 

    韩国: master_track_kr, kr_next_version, kr_next_develop, kr_master, kr_develop

    越南: master_track_vnm, vnm_next_version, vnm_next_develop, vnm_master, vnm_develop

    泰国: master_track_th, thai_next_version, thai_next_develop, thai_master, thai_develop

    下面采用了后面5个通用分支名来说明, master_track_xx, xx_next_version, xx_next_develop, xx_master, xx_develop

  • master_track_xx
    • 该分支提供三个功能，第一：从国服 master 上去版本, merge 相应的 tag；第二：针对给平台版本做一些持久化修改，比如去掉敏感词库, 关闭答题系统，开启跨服聊天, 注册字符限制，屏蔽强化祝福等；第三：作为 xx_next_version 分支的来源
  • xx_next_version
    • 该分支立足于：预备当前规划的下一次的版本更新, 非采用屏蔽模式的分期更新， 该分支需要从 master_track_xx 上取下一次需要更新的版本。且应该在该分支上做只针对当前版本需要做的特殊化处理，所以建议每次版本更新采用覆盖的方式, 即：每次版本更新先清空该分支下文件，再拷贝 master_track_xx 上的文件到该分支下。鉴于翻译的时间消耗，应尽早规划下一次的版本更新，同时也体现该分支的必要性。
  • xx_next_develop
    • 该分支立足于：预备当前规划的下一次的采用屏蔽模式的分期更新， 该分支需要根据分期规划开放相应的功能模块等。同时每个版本的分期性屏蔽处理应该在该分支上进行。如果是版本更新，应该 merge 分支 xx_next_version。该分支对应测试服的NEXT服。
  • xx_develop
    • 该分支相当于国服的 develop，作为针对线上服的测试服。每次版本(分期)更新应该 merge 分支 xx_next_develop。该分支对应测试服的 0 服
  • xx_master
    • 该分支相当于国服的 master，直供线上服，新开服。每次版本(分期)更新应该 merge 分支 xx_develop。

关于服务端翻译工作

  • 台湾
  • 韩国/越南/泰国

翻译工作 —— 台湾

  • 1:更新ldata.hrl文件
    • 通过 ldata:find/0 会更新 ldata.hrl 文件,
  • 2:更新源码中数据文件(文件名包括data)
    • 确认 ldata.hrl 文件没有问题后，通过 ldata:replace/0 将数据文件中的简体中文上加上 多语言函数头 language:get/1
  • 3:进行一次编译
    • 将一些明显的问题处理了
  • 4:导入现有繁化好了的数据到 dets 文件
    • 通过 lserver:import/1 导入现有繁化好了的翻译数据(server_done.xml)
  • 5:重置翻译数据版本号
    • 通过 lserver:set_ver/1, 这里传入参数为 0
  • 6:重新抽取简体
    • 通过 lserver:ext/0 重新抽取语言包
  • 7:导出新增未繁化语言包
    • 通过 lserver:export/1 这里参数是 undone, 导出的文件是 server_undone.xml
  • 8:繁化新增语言包
    • 通过 office 简转繁，将 cn 列的简体拷贝到 tr 列，在单独将 tr 列的进行简转繁
  • 9:修改特殊翻译内容
    • 用 vim 打开 server_undone.xml， 通过正则将 繁体中的 夢幻飛仙 改成 萌飛仙，充值 改成 儲值，首充 改成首儲，如下:
    %s/夢幻飛仙/萌飛仙/g
    %s/\(.*\)<tr>\(.*\)充值\(.*\)/\1<tr>\2儲值\3/g
    %s/\(.*\)<tr>\(.*\)首充\(.*\)/\1<tr>\2首儲\3/g
  • 10:导入繁化好了的新增语言包
    • 通过 lserver:import/1 导入新增繁化好了的翻译数据(server_undone.xml)
  • 11:生成新的erl语言包
    • 通过 lserver:gen/0 会生成新的 language.erl 文件
  • 12:导出新的语言包
    • 通过 lserver:export/1 这里参数是 done, 导出的文件是 server_done.xml
  • 13:检测language.erl 文件
    • 根据错误，并修正相应的文件
  • 14:提交语言包
    • server_done.xml , language.erl 等
